What is KICS?
Keeping Infrastructure as Code Secure (KICS) is an open-source static analysis engine by Checkmarx. It scans Terraform, CloudFormation, Kubernetes, Ansible and more — finding security misconfigurations before they reach production.

Checkov is the Bridgecrew/Prisma Cloud equivalent and shares most check IDs. Both are widely used in CI/CD pipelines for policy-as-code enforcement.
⚡
How does it scan?
KICS reads your .tf source files statically — no AWS API calls, no plan needed. It maps resource blocks to a library of Rego-based queries and flags patterns that match known misconfigurations.

This means it catches issues at authoring time, before terraform plan or apply runs. In GitLab CI you run it as a SAST job on every merge request.
🏦
Why it matters for financial services
KMS and IAM misconfigurations are among the most frequently cited findings in cloud security audits. Many map directly to regulatory controls:

PCI-DSS 3.6.4 — key rotation · CIS AWS 1.16 — no admin wildcards · SOC 2 CC6.3 — least privilege · PCI-DSS 7.1.2 — minimal access
How KICS fits into a CI/CD pipeline
01 / AUTHOR
Write Terraform
Developer writes .tf files and opens a merge request against the trunk branch.
02 / SCAN
KICS SAST job
GitLab CI pipeline runs kics scan -p . --report-formats json on the changed files. No AWS credentials needed.
03 / GATE
Block or warn
Critical findings block the MR from merging. High findings raise a warning. Findings are published to the GitLab Security Dashboard.
04 / FIX
Remediate & re-scan
Developer applies the fix shown in this reference, pushes a new commit, and the pipeline re-runs automatically.

How each pipeline stage works
Stage 01 · validate
Terraform format & validate
Runs terraform fmt -check and terraform validate. No AWS credentials needed — purely static HCL parsing. Fails fast on syntax errors before spending time on security scans.
Stage 02 · sast
KICS IaC SAST scan
Runs kics scan --path . --fail-on CRITICAL,HIGH. KICS reads every .tf file statically — no AWS API calls, no plan needed. Exits non-zero if Critical or High findings are detected.
Stage 03 · plan
Terraform plan (speculative)
Runs terraform plan -out=tfplan against the target account using OIDC federation. Only runs if the SAST stage passed. Plan output is saved as an artifact for human review in the MR.
Stage 04 · security-gate
MR merge gate decision
Parses the KICS JSON report artifact. Critical findings block the MR from merging via GitLab's needs: dependency — the gate job fails, preventing the approval button from activating.
Job definition
validate job
1tf-validate:
2 stage: validate
3 image: hashicorp/terraform:1.7
4 script:
5 - terraform init -backend=false
6 - terraform fmt -check -recursive
7 - terraform validate
8 rules:
9 - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
Why this comes first
  • KICS can still scan invalid HCL but produces noisy false results. Validating first ensures findings are real.
  • -backend=false skips state backend init — no AWS credentials needed at this stage.
  • Format check enforces consistent code style before review. Fails early, saves reviewer time.
  • Exit code 0 = passes and unblocks the sast stage. Non-zero = pipeline stops here.
KICS job definition
kics-iac-sast job
1kics-iac-sast:
2 stage: sast
3 needs: [tf-validate]
4 image: checkmarx/kics:latest
5 script:
6 - kics scan \
7 --path . \
8 --report-formats json,html \
9 --output-path reports/ \
10 --fail-on CRITICAL,HIGH
11 artifacts:
12 reports:
13 sast: reports/kics-results.json
14 when: always
Key flags explained
  • --path . — scans all .tf files in the repo recursively
  • --report-formats json,html — JSON consumed by the gate job; HTML uploaded for human review
  • --fail-on CRITICAL,HIGH — exits non-zero only for these severities. Medium/Low are informational.
  • --output-path reports/ — artifacts directory; passed to next stage via GitLab artifacts
  • when: always on artifacts — report is uploaded even when the job fails, so reviewers can see what KICS found
  • No AWS credentials needed — purely static file analysis

OIDC federation — no static keys
  • id_tokens block generates a short-lived OIDC JWT — GitLab CI's native way to authenticate to AWS without storing access keys
  • The AWS IAM role trust policy must match the GitLab OIDC issuer and the specific project/branch claims
  • needs: [kics-iac-sast] creates a hard dependency — if KICS found Critical/High issues, this stage never runs
  • The plan is a speculative dry-run — no infrastructure is created at this stage
  • Plan artifact is used by the security-gate job and displayed in the MR for reviewer inspection
Gate job definition
security-gate job
1security-gate:
2 stage: security-gate
3 needs:
4 - kics-iac-sast
5 - tf-plan
6 image: python:3.12-slim
7 script:
8 - python3 scripts/kics_gate.py \
9 --report reports/kics-results.json \
10 --fail-on CRITICAL
11 allow_failure: false
How the gate blocks the MR
  • The gate script parses kics-results.json and counts findings by severity
  • If Critical count > 0: sys.exit(1) — the job fails with a non-zero exit code
  • GitLab marks the MR as "pipeline failed" — the Merge button stays disabled until the pipeline passes
  • allow_failure: false (default) ensures failure propagates to the MR status
  • High findings log a warning but do not exit non-zero — they appear as informational comments on the MR via the SAST report artifact
  • The gate job also posts a summary comment to the MR via the GitLab API using $CI_MERGE_REQUEST_IID

CKV_AWS_111 · aws_kms_key · aws_kms_key_policy
Wildcard principal with no condition
Fires when Principal = "*" appears in an Allow statement with no Condition block. Grants any AWS identity globally — including identities in other accounts. Also fires for Principal = { AWS = "*" }.
Critical
Diff
KICS output
Condition workaround
✗ Vulnerable
1Statement = [{
2 Effect = "Allow"
3 Principal = "*"
4 Action = ["kms:*"]
5 Resource = "*"
6 # no Condition block
7}]
✓ Fixed — scoped principal
1Statement = [{
2 Effect = "Allow"
3 Principal = { AWS = [
4 "arn:aws:iam::${local.account_id}:root"
5 ]}
6 Action = local.kms_use_actions
7 Resource = "*"
8}]
What triggers it
  • Allow + "*" principal + no Condition — all three together
  • Also { AWS = "*" } with same pattern
  • Applies to both inline policy on aws_kms_key and the standalone aws_kms_key_policy resource
What does NOT trigger it
  • Specific role/user/account ARN as Principal
  • Service principal: { Service = "logs.amazonaws.com" }
  • Wildcard Principal WITH a Condition block present (see workaround tab)
Check: CKV_AWS_111: "Ensure IAM policies do not have statements with admin permissions" FAILED for resource: aws_kms_key.bad File: main.tf:1-7 2 | Effect = "Allow" 3 | Principal = "*" 4 | Action = ["kms:*"] Passed checks: 0, Failed checks: 1, Skipped checks: 0
⚠If you genuinely need Principal = "*" (e.g. cross-account service access), KICS accepts it only when a Condition block is also present to restrict scope.
✓ Accepted — wildcard principal + condition
1Statement = [{
2 Effect = "Allow"
3 Principal = "*"
4 Action = local.kms_use_actions
5 Resource = "*"
6 Condition = {
7 StringEquals = {
8 "kms:CallerAccount" =
9 data.aws_caller_identity.current.account_id
10 }
11 }
12}]
CKV_AWS_356 · aws_kms_key · aws_kms_key_policy
Wildcard actions in key policy
Fires even when Principal is scoped, if Action contains wildcards like kms:* or kms:Delete*. Requires splitting the policy into two statements — one admin, one use — each with an explicit action list.
Critical
Diff
Action allowlists
✗ Vulnerable
1 Principal = { AWS = module.app.role_arn }
2 Action = ["kms:*"]
3 # includes DeleteKey, PutKeyPolicy…
✓ Fixed — named locals
1 Principal = { AWS = module.app.role_arn }
2 Action = local.kms_use_actions
3 # Encrypt, Decrypt, GenerateDataKey…
locals.tf — curated action sets

6 "kms:GenerateDataKeyWithoutPlaintext", "kms:DescribeKey"
7 ]
8 # Infra/admin role — control-plane
9 kms_admin_actions = [
10 "kms:Create*", "kms:Describe*", "kms:Enable*",

CKV2_AWS_6 · aws_kms_key
No key policy defined — default policy applied
When the policy block is omitted from aws_kms_key, AWS silently applies a default policy granting the account root full kms:*. KICS skips CKV_AWS_111 (nothing to scan) but CKV2_AWS_6 catches the omission directly.
Critical
⚠Counterintuitive: omitting the policy block bypasses CKV_AWS_111 but immediately triggers CKV2_AWS_6. The AWS default policy at runtime is equivalent to Principal="*", Action="kms:*" — equally dangerous, just invisible to the static scanner.
✗ No policy — CKV2_AWS_6 fires
1resource "aws_kms_key" "bad" {
2 description = "app key"
3 enable_key_rotation = true
4 # policy block omitted entirely
5 # CKV_AWS_111 → skipped (nothing to scan)
6 # CKV2_AWS_6 → FAILS
7}
✓ Explicit policy required
1resource "aws_kms_key" "good" {
2 description = "app key"
3 enable_key_rotation = true
4 policy = data.aws_iam_policy_document
5 .kms_policy.json
6 # CKV2_AWS_6 → PASSES
7}
Finding matrix — no policy block
  • CKV_AWS_111 → skipped (no block for the scanner to evaluate)
  • CKV2_AWS_6 → FAILS
  • CKV_AWS_7 and CKV_AWS_227 → still evaluated (separate attributes, unaffected)
AWS default key policy (runtime, invisible to KICS)
  • Principal: account root ARN (arn:aws:iam::ACCOUNT:root)
  • Action: kms:* · Resource: *
  • Functionally identical to the pattern that triggers CKV_AWS_111 and CKV_AWS_356 when written explicitly
CKV_AWS_7 · aws_kms_key
Key rotation not enabled
enable_key_rotation defaults to false. Omitting it fails this check. Required by PCI-DSS 3.6.4, CIS AWS Benchmark 3.8, and SOC 2 CC9.1.
High
✗ Vulnerable — default is false
1resource "aws_kms_key" "bad" {
2 description = "app key"
3 # enable_key_rotation not set → false
4}
✓ Fixed — one line
1resource "aws_kms_key" "good" {
2 description = "app key"
3 enable_key_rotation = true
4}
Rotation behaviour
  • AWS rotates the backing key material automatically every 365 days
  • Key ARN and aliases are unchanged — no re-encryption needed for existing data
  • Old material is retained indefinitely for decryption of existing ciphertext
Not supported for
  • Asymmetric KMS keys (RSA, ECC) — manage manually
  • HMAC keys
  • CloudHSM custom key stores — manage rotation via HSM tooling

CKV_AWS_356 · aws_iam_policy · aws_iam_policy_document
IAM policy uses Resource = "*" for KMS actions
Flags policies where KMS actions are granted on "*" rather than a specific key ARN. Important caveat: several KMS actions require "*" by AWS design — KICS has a documented false-positive for these.
Critical
ℹKnown false positive (Checkov issues #5148, #5305): kms:GenerateDataKey or kms:Decrypt with Resource = "*" and a Condition block still fire CKV_AWS_356 in some scanner versions. Actions like kms:ListKeys require "*" per AWS — use a skip annotation.
Diff
Skip annotation
✗ Flagged
1 statement {
2 actions = ["kms:Decrypt"]
3 resources = ["*"]
4 }
✓ Scoped to key ARN
1 statement {
2 actions = ["kms:Decrypt"]
3 resources = [aws_kms_key.this.arn]
4 }
Actions requiring Resource = "*" (use skip)
  • kms:ListKeys, kms:ListAliases
  • kms:CreateKey, kms:GenerateRandom
Actions safely scoped to key ARN
  • kms:Decrypt, kms:Encrypt, kms:DescribeKey
  • kms:GenerateDataKey, kms:ReEncrypt*, kms:PutKeyPolicy
Skip annotation for actions that require "*"
1data "aws_iam_policy_document" "kms_list" {
2 #checkov:skip=CKV_AWS_356:kms:ListKeys requires Resource="*" per AWS documentation
3 statement {
4 effect = "Allow"
5 actions = ["kms:ListKeys", "kms:ListAliases"]
6 resources = ["*"]
7 }
8}
Skip annotation rules
  • Justification is mandatory — the scanner records it in the HTML/JSON report
  • Skips should be reviewed periodically; add a ticket reference if your org policy requires it
  • Never skip CKV_AWS_49 or CKV_AWS_1 — those indicate genuinely unsafe policies

CKV_AWS_1 · aws_iam_policy · aws_iam_policy_document · aws_iam_role_policy
Full administrative privileges — Action:* Resource:*
The most severe IAM finding. Fires when any Allow statement grants Action = "*" and Resource = "*" simultaneously. This is equivalent to attaching AdministratorAccess inline — the bearer can do anything in the AWS account.
Critical
Diff
KICS output
Trigger conditions
✗ Vulnerable
1data "aws_iam_policy_document" "bad" {
2 statement {
3 effect = "Allow"
4 actions = ["*"]
5 resources = ["*"]
6 }
7}
✓ Fixed — scoped service actions
1data "aws_iam_policy_document" "good" {
2 statement {
3 effect = "Allow"
4 actions = ["s3:GetObject", "s3:PutObject",
5 "ec2:DescribeInstances"]
6 resources = [aws_s3_bucket.this.arn]
7 }
8}
Compliance mapping
  • CIS AWS Benchmark 1.16 — Ensure no admin-privilege IAM policies exist
  • SOC 2 CC6.3 — Logical access limited to minimum necessary
  • PCI-DSS 7.1.2 — Restrict access to individuals whose job requires it
Also fires for these resource types
  • Inline JSON in aws_iam_policy.policy
  • Heredoc (<<EOF) policies with wildcard statements
  • aws_iam_role_policy inline role policies
  • Related check: CKV2_AWS_40 covers the iam:* variant
Check: CKV_AWS_1: "Ensure IAM policies that allow full '*' administrative privileges are not created" FAILED for resource: data.aws_iam_policy_document.bad 4 | actions = ["*"] 5 | resources = ["*"] Passed checks: 0, Failed checks: 1, Skipped checks: 0
Exact trigger conditions
  • Effect = "Allow" AND Action = "*" AND Resource = "*" — all three together
  • Does NOT fire if Effect = "Deny" — a deny-all is a valid lockdown pattern
  • Does NOT fire for service-scoped wildcards like "s3:*" — only the bare "*" triggers CKV_AWS_1; CKV_AWS_49 covers service wildcards
  • Does NOT fire if Resource is a specific ARN even with Action = "*" — though CKV_AWS_49 still catches the action wildcard separately

109 vs 110 — what's the difference
  • CKV_AWS_109 — direct policy mutation (rewriting another role's policy directly)
  • CKV_AWS_110 — indirect escalation (actions that can be chained to reach higher access)
  • Example chain: iam:CreatePolicyVersion → create permissive version of existing policy → iam:SetDefaultPolicyVersion → activate it silently
Permissions boundary mitigation
  • AWS Permissions Boundaries cap what a role can grant even if it can write policies
  • Require iam:PermissionsBoundary condition on all iam:CreateRole actions
  • Enforce via SCP at org level for defence-in-depth
Known privilege escalation action vectors (trigger CKV_AWS_110)
  • sts:AssumeRole on "*" — assume any role including admin roles
  • iam:CreatePolicyVersion — overwrite an existing policy to grant full admin
  • iam:SetDefaultPolicyVersion — activate a previously created permissive version
  • iam:PassRole on "*" — pass a privileged role to Lambda/EC2 to execute arbitrary code under that role
  • iam:CreateRole + iam:AttachRolePolicy without path constraint — create a new admin role
  • lambda:UpdateFunctionCode on a function running as an admin role — replace code to exfiltrate credentials
CKV2_AWS_40 · aws_iam_policy · aws_iam_role_policy
IAM managed policy grants full IAM privileges (iam:*)
The v2 companion to CKV_AWS_1. Targets managed policies (aws_iam_policy) and inline role policies (aws_iam_role_policy). Fires when the policy grants iam:* on Resource = "*" — full IAM control without necessarily being a full admin policy.
