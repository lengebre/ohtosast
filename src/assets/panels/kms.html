CKV_AWS_111 · aws_kms_key · aws_kms_key_policy
Wildcard principal with no condition
This finding appears when Principal = "*" is allowed with no Condition. Any AWS identity in any account could use the key. The same rule applies to Principal = { AWS = "*" }.
Critical
Diff
KICS output
Condition workaround
✗ Vulnerable
1Statement = [{
2 Effect = "Allow"
3 Principal = "*"
4 Action = ["kms:*"]
5 Resource = "*"
6 # no Condition block
7}]
✓ Fixed: scoped principal
1Statement = [{
2 Effect = "Allow"
3 Principal = { AWS = [
4 "arn:aws:iam::${local.account_id}:root"
5 ]}
6 Action = local.kms_use_actions
7 Resource = "*"
8}]
What triggers it
  • Allow + "*" principal + no Condition: all three together
  • Also { AWS = "*" } with same pattern
  • Applies to both inline policy on aws_kms_key and the standalone aws_kms_key_policy resource
What does NOT trigger it
  • Specific role/user/account ARN as Principal
  • Service principal: { Service = "logs.amazonaws.com" }
  • Wildcard Principal WITH a Condition block present (see workaround tab)
Check: CKV_AWS_111: "Ensure IAM policies do not have statements with admin permissions" FAILED for resource: aws_kms_key.bad File: main.tf:1-7 2 | Effect = "Allow" 3 | Principal = "*" 4 | Action = ["kms:*"] Passed checks: 0, Failed checks: 1, Skipped checks: 0
⚠If you genuinely need Principal = "*" (e.g. cross-account service access), KICS accepts it only when a Condition block is also present to restrict scope.
✓ Accepted: wildcard principal + condition
1Statement = [{
2 Effect = "Allow"
3 Principal = "*"
4 Action = local.kms_use_actions
5 Resource = "*"
6 Condition = {
7 StringEquals = {
8 "kms:CallerAccount" =
9 data.aws_caller_identity.current.account_id
10 }
11 }
12}]
CKV_AWS_356 · aws_kms_key · aws_kms_key_policy
Wildcard actions in key policy
This can still fire when the principal is narrowed but Action uses wildcards such as kms:* or kms:Delete*. Fix it by using two statements: one for admins with tight actions, one for app roles with a fixed list of KMS actions (no wildcards).
Critical
Diff
Action allowlists
✗ Vulnerable
1 Principal = { AWS = module.app.role_arn }
2 Action = ["kms:*"]
3 # includes DeleteKey, PutKeyPolicy…
✓ Fixed: named locals
1 Principal = { AWS = module.app.role_arn }
2 Action = local.kms_use_actions
3 # Encrypt, Decrypt, GenerateDataKey…
locals.tf: curated action sets
1locals {
2 # App role: data-plane only (use these for CKV_AWS_356)
3 kms_use_actions = [
4 "kms:Decrypt", "kms:Encrypt",
5 "kms:ReEncrypt*", "kms:GenerateDataKey",
6 "kms:GenerateDataKeyWithoutPlaintext", "kms:DescribeKey"
7 ]
8 # Infra/admin role: control-plane
9 kms_admin_actions = [
10 "kms:Create*", "kms:Describe*", "kms:Enable*",
11 "kms:List*", "kms:Put*", "kms:Update*",
12 "kms:Disable*", "kms:Delete*",
13 "kms:ScheduleKeyDeletion", "kms:CancelKeyDeletion"
14 ]
15}
CKV2_AWS_6 · aws_kms_key
No key policy defined: default policy applied
When the policy block is omitted from aws_kms_key, AWS silently applies a default policy granting the account root full kms:*. KICS skips CKV_AWS_111 (nothing to scan) but CKV2_AWS_6 catches the omission directly.
Critical
⚠Skipping the policy block avoids CKV_AWS_111 but triggers CKV2_AWS_6. At runtime AWS still applies a very open default (similar to Principal="*" and Action="kms:*"). The scanner cannot see that default, but the risk is real.
✗ No policy: CKV2_AWS_6 fires
1resource "aws_kms_key" "bad" {
2 description = "app key"
3 enable_key_rotation = true
4 # policy block omitted entirely
5 # CKV_AWS_111 → skipped (nothing to scan)
6 # CKV2_AWS_6 → FAILS
7}
✓ Explicit policy required
1resource "aws_kms_key" "good" {
2 description = "app key"
3 enable_key_rotation = true
4 policy = data.aws_iam_policy_document
5 .kms_policy.json
6 # CKV2_AWS_6 → PASSES
7}
Finding matrix: no policy block
  • CKV_AWS_111 → skipped (no block for the scanner to evaluate)
  • CKV2_AWS_6 → FAILS
  • CKV_AWS_7 and CKV_AWS_227 → still evaluated (separate attributes, unaffected)
AWS default key policy (runtime, invisible to KICS)
  • Principal: account root ARN (arn:aws:iam::ACCOUNT:root)
  • Action: kms:* · Resource: *
  • Functionally identical to the pattern that triggers CKV_AWS_111 and CKV_AWS_356 when written explicitly
CKV_AWS_7 · aws_kms_key
Key rotation not enabled
enable_key_rotation defaults to false. Omitting it fails this check. Required by PCI-DSS 3.6.4, CIS AWS Benchmark 3.8, and SOC 2 CC9.1.
High
✗ Vulnerable: default is false
1resource "aws_kms_key" "bad" {
2 description = "app key"
3 # enable_key_rotation not set → false
4}
✓ Fixed: one line
1resource "aws_kms_key" "good" {
2 description = "app key"
3 enable_key_rotation = true
4}
Rotation behaviour
  • AWS rotates the backing key material automatically every 365 days
  • Key ARN and aliases are unchanged: no re-encryption needed for existing data
  • Old material is retained indefinitely for decryption of existing ciphertext
Not supported for
  • Asymmetric KMS keys (RSA, ECC): manage manually
  • HMAC keys
  • CloudHSM custom key stores: manage rotation via HSM tooling
CKV_AWS_227 · aws_kms_key
KMS key explicitly disabled
is_enabled defaults to true so this only fires when you set it to false. A disabled key blocks all encrypt/decrypt operations for every service referencing it.
High
✗ Flagged

2 description = "app key"
3 is_enabled = false
4}

2 description = "app key"
