CKV_AWS_1 · aws_iam_policy · aws_iam_policy_document · aws_iam_role_policy
Full admin: Action * on Resource *
The worst-case IAM pattern: an Allow with Action = "*" and Resource = "*" together. That is effectively full administrator access to the whole account.
Critical
Diff
KICS output
Trigger conditions
✗ Vulnerable
1data "aws_iam_policy_document" "bad" {
2 statement {
3 effect = "Allow"
4 actions = ["*"]
5 resources = ["*"]
6 }
7}
✓ Fixed: scoped service actions
1data "aws_iam_policy_document" "good" {
2 statement {
3 effect = "Allow"
4 actions = ["s3:GetObject", "s3:PutObject",
5 "ec2:DescribeInstances"]
6 resources = [aws_s3_bucket.this.arn]
7 }
8}
Compliance mapping
  • CIS AWS Benchmark 1.16: Ensure no admin-privilege IAM policies exist
  • SOC 2 CC6.3: Logical access limited to minimum necessary
  • PCI-DSS 7.1.2: Restrict access to individuals whose job requires it
Also fires for these resource types
  • Inline JSON in aws_iam_policy.policy
  • Heredoc (<<EOF) policies with wildcard statements
  • aws_iam_role_policy inline role policies
  • Related check: CKV2_AWS_40 covers the iam:* variant
Check: CKV_AWS_1: "Ensure IAM policies that allow full '*' administrative privileges are not created" FAILED for resource: data.aws_iam_policy_document.bad 4 | actions = ["*"] 5 | resources = ["*"] Passed checks: 0, Failed checks: 1, Skipped checks: 0
Exact trigger conditions
  • Effect = "Allow" AND Action = "*" AND Resource = "*": all three together
  • Does NOT fire if Effect = "Deny": a deny-all is a valid lockdown pattern
  • Does NOT fire for service-scoped wildcards like "s3:*": only the bare "*" triggers CKV_AWS_1; CKV_AWS_49 covers service wildcards
  • Does NOT fire if Resource is a specific ARN even when Action = "*" (CKV_AWS_49 may still flag the action wildcard)
CKV_AWS_107 · aws_iam_policy · aws_iam_policy_document
IAM policy allows credentials exposure
Fires when a policy grants actions that return long-lived credentials or token material without a constraining Condition. Canonical triggers: iam:CreateAccessKey, iam:UpdateAccessKey, sts:GetFederationToken on Resource = "*".
Critical

Check: CKV_AWS_107: "Ensure IAM policies does not allow credentials exposure" FAILED for resource: data.aws_iam_policy_document.bad 3 | actions = ["iam:CreateAccessKey", "iam:UpdateAccessKey"] 5 | resources = ["*"] Passed checks: 0, Failed checks: 1, Skipped checks: 0
Actions triggering CKV_AWS_107 when Resource = "*" without Condition
  • iam:CreateAccessKey: creates long-lived keys for any user
  • iam:UpdateAccessKey: activates/deactivates access keys on any user
  • iam:CreateLoginProfile / iam:UpdateLoginProfile: sets/changes console password
  • sts:GetFederationToken: issues federated session tokens
  • sts:AssumeRoleWithWebIdentity / sts:AssumeRoleWithSAML
  • Prefer IAM roles + instance profiles or OIDC federation over long-lived access keys wherever possible
CKV_AWS_108 · aws_iam_policy · aws_iam_policy_document
IAM policy allows data exfiltration
Fires when data-read actions on sensitive services (s3:GetObject, secretsmanager:GetSecretValue, ssm:GetParameter*) are granted on Resource = "*" without a Condition. A role with this policy could read every secret, S3 object, and SSM parameter in the account.
Critical
Diff
Trigger actions
✗ Vulnerable

2 effect = "Allow"
3 actions = ["secretsmanager:GetSecretValue",
4 "ssm:GetParameter", "s3:GetObject"]
5 resources = ["*"]
6 }
✓ Fixed: scoped to app prefix
1 statement {
2 effect = "Allow"
3 actions = ["secretsmanager:GetSecretValue"]
4 resources = [
5 "arn:aws:secretsmanager:eu-west-2:${local.account_id}"

7 ]

Actions triggering CKV_AWS_108 (data exfiltration) when Resource = "*"
  • s3:GetObject, s3:GetObjectVersion
  • secretsmanager:GetSecretValue
  • ssm:GetParameter, ssm:GetParameters, ssm:GetParametersByPath, ssm:GetParameterHistory
  • rds:CopyDBSnapshot, rds:CreateDBSnapshot
  • glacier:GetObject, glacier:InitiateJob
  • Fix: always scope resources to specific ARN prefixes per app/environment
CKV_AWS_109 · aws_iam_policy · aws_iam_policy_document
IAM policy allows permissions management without constraints
Triggers when powerful IAM APIs (for example iam:PutRolePolicy, iam:AttachRolePolicy, iam:CreatePolicy) are allowed on Resource = "*" with no Condition. A caller could change other roles and escalate to full admin.
Critical
Diff
Trigger actions
✗ Vulnerable
1 statement {
2 effect = "Allow"
3 actions = ["iam:PutRolePolicy",
4 "iam:AttachRolePolicy"]
5 resources = ["*"]
6 # can mutate any role's policy
7 }
✓ Fixed: path + PolicyARN condition
1 statement {
2 effect = "Allow"
3 actions = ["iam:PutRolePolicy"]
4 resources = ["arn:aws:iam::${local.account_id}:role/app/*"]
5 condition {
6 test = "StringLike"
7 variable = "iam:PolicyARN"
8 values = ["arn:aws:iam::${local.account_id}:policy/app/*"]
9 }
10 }
Actions triggering CKV_AWS_109 when Resource = "*" and no Condition
  • iam:PutRolePolicy: writes inline policy to any role
  • iam:AttachRolePolicy / iam:AttachUserPolicy / iam:AttachGroupPolicy
  • iam:CreatePolicy, iam:CreatePolicyVersion, iam:SetDefaultPolicyVersion
  • iam:PutGroupPolicy, iam:PutUserPolicy
  • iam:AddUserToGroup: can silently add user to an admin group
  • Fix: scope resources to an IAM path prefix and add a Condition on iam:PolicyARN or iam:PermissionsBoundary
CKV_AWS_110 · aws_iam_policy · aws_iam_policy_document
IAM policy allows privilege escalation
Different from CKV_AWS_109. Here the risk is chaining steps that look small alone (for example sts:AssumeRole on Resource = "*", which lets someone assume powerful roles including admin).
Critical
Diff
Escalation vectors
✗ Vulnerable: unconstrained assume role
1 statement {
2 effect = "Allow"
3 actions = ["sts:AssumeRole"]
4 resources = ["*"]
5 # can assume ANY role inc. admin
6 }
✓ Fixed: explicit role ARNs only
1 statement {
2 effect = "Allow"
3 actions = ["sts:AssumeRole"]
4 resources = [
5 "arn:aws:iam::${local.account_id}:role/app-deploy",
6 "arn:aws:iam::${local.account_id}:role/app-read"
7 ]
8 }
CKV_AWS_109 vs CKV_AWS_110
  • CKV_AWS_109: changing another role's policy in one step
  • CKV_AWS_110: smaller permissions that can be combined to reach admin-level access
  • Example chain: iam:CreatePolicyVersion → create permissive version of existing policy → iam:SetDefaultPolicyVersion → activate it silently
Permissions boundary mitigation
  • AWS Permissions Boundaries cap what a role can grant even if it can write policies
  • Require iam:PermissionsBoundary condition on all iam:CreateRole actions
  • Enforce via SCP at org level for defence-in-depth
Known privilege escalation action vectors (trigger CKV_AWS_110)
  • sts:AssumeRole on "*": assume any role including admin roles
  • iam:CreatePolicyVersion: overwrite an existing policy to grant full admin
  • iam:SetDefaultPolicyVersion: activate a previously created permissive version
  • iam:PassRole on "*": pass a privileged role to Lambda/EC2 to execute arbitrary code under that role
  • iam:CreateRole + iam:AttachRolePolicy without path constraint: create a new admin role
  • lambda:UpdateFunctionCode on a function running as an admin role: replace code to exfiltrate credentials
CKV2_AWS_40 · aws_iam_policy · aws_iam_role_policy
IAM managed policy grants full IAM privileges (iam:*)
Related to CKV_AWS_1 but looks at iam:* on Resource = "*" in managed or inline role policies. That gives full control over IAM even if other AWS services are not wide open.
Critical
⚠CKV_AWS_1 flags Action = "*" (every AWS service). CKV2_AWS_40 flags iam:* on all resources (full IAM API access). They are different checks and both can alert on the same policy.
✗ Vulnerable: iam:* on managed policy
1resource "aws_iam_policy" "bad" {
2 name = "bad-iam-policy"
3 policy = jsonencode({
4 Statement = [{ Effect = "Allow"
5 Action = ["iam:*"]
6 Resource = "*" }]
7 })
8}
✓ Fixed: explicit read-only IAM actions
1resource "aws_iam_policy" "good" {
2 name = "app-iam-read"
3 policy = jsonencode({
4 Statement = [{ Effect = "Allow"
5 Action = ["iam:GetRole", "iam:ListRoles",
6 "iam:GetPolicy", "iam:ListPolicies"]
7 Resource = "*" # list ops require "*"
8 }]
9 })
